Naming Scheme for Collision Data Generation

If collision data is desired, proxy box geometry needs to be created. This is true for setting collision data using Alternate Bounding Volumes, Oriented Bounding Boxes, or even Triangles. In order for the proxy geometry to be identified by the CreateABV plug-in and converted into collision data, the proxy geometry must be named in a particular way.

For all the types, the first portion of the proxy geometry name must be "NDLCD" (NDL Collision Data). This says that some sort of collision data is desired. Next, a space followed by a 2 letter code is required, for example "NDLCD SN". The first letter represents the type of collision and the second letter represents the propagation control method collision propagation method. The letter codes are as listed below:

Collision Type Letter Codes

  • S = Sphere
  • C = Capsule
  • B = Box
  • H = Halfspace
  • U = Union
  • T = Triangle
  • O = OBB
  • N = No Test
  • W = Use bounding sphere

Propagation Mode Letter Codes

  • A = Always Propagate
  • N = Never Propagate
  • S = Propagate on a Successful Collision
  • F = Propagate on a Failed Collision

The rest of the name is ignored, but may be used by the artist or application for whatever purpose is necessary. For example, a box with the name "NDLCD SN My Sphere Bound" would generate a spherical ABV that would be attached by way of collision data to the box's parent in the scene hierarchy. The propagation mode would be such that collision would never propagate.

Triangle Collisions and Mesh Profiles

Meshes used for Triangle collision types have special requirements on their data. The sample Mesh Profile named PosNormPicking.xml sets the appropriate access for both triangle collisions and picking, and any triangle collisions should use that mesh profile or another like it. The profile must be extended if color or texture information is required from picking.
